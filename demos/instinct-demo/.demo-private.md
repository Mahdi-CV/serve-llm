# PRIVATE - personal cheat sheet

This is the active demo folder. `~/vp-demo` is kept as a backup (older approach
that used a container named after the model).

## What is set up

- A single general-purpose vLLM container `vllm-openai-rocm-ctr` runs as standing
  infrastructure on the server. It is idle (sleep entrypoint) and stays up.
- Serving a model = launching `vllm serve` inside that container via `docker exec`.
- The vLLM compile cache lives on the host disk at `/mnt/scratch/vllm_cache` and is
  mounted in, so graphs survive restarts and even container recreation.
- Between takes we just kill the `vllm serve` process; the container stays up. The
  next run relaunches the model from warm cache (~90s) and shows the engine warmup.

## The prompt to say on camera

> run Qwen3.5 27B on my Instinct server at 134.199.193.61 with vLLM and give me the endpoint

## Flow (every take)

1. Stage the box from the laptop:

       cd ~/instinct-demo && ./stage.sh

   This ensures the container is up and kills any running model process.

2. Open Claude in this folder:

       claude --dangerously-skip-permissions

3. Say the prompt. Endpoint + table in roughly 2 minutes.

## Reset between takes

Run `./stage.sh` again before the next take.

## Timing

- First-ever launch after the cache mount changed: ~243s (one-time, cache populate).
- Warm relaunch after a process kill: ~90s + Claude orchestration. This is the demo path.

## What is honest to say if asked

- The weights are cached locally (no download).
- We keep a general-purpose vLLM serving container running on the box; serving a
  model starts it inside that container. The ~2 min is the engine warming up
  (loading weights into HBM, replaying compiled graphs, allocating the KV cache).
- A brand-new model from a cold box takes longer; we pre-stage so we stay in the
  time budget.

## Backup

If anything misbehaves, `~/vp-demo` + its `stage.sh` still work (per-model container,
`docker start` path). The two share port 8002, so only run one at a time.
